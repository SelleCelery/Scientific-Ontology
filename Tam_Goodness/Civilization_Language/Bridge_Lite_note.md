# Bridge Lite: 基底整合と3核保存による構造保存的文章転写の試み

## 概要

本稿は、高密度かつ多位相的な自己記述文を対象に、通常の要約や平文化では失われやすい構造的核を、できるだけ保持したまま別の説明レジームへ移すための最小枠組みとして **Bridge Lite** を報告するものである。今回の試作では、`plain`、`basis_on`、`bridge_on` の三モードを用い、単一のテスト文に対して逐次比較を行った。比較の結果、`plain` は入力文内部の骨格整理、`basis_on` は基底語彙への整列、`bridge_on` は不変核と変換可能構造の分離という、異なる操作として機能した。特に `bridge_on` では、「目的」「方法」「提示プロトコル」の三核を明示的に扱うことにより、文章の内容だけでなく、その提示条件まで含めた転写が可能になることが示された。

## 問題設定

対象とした文章は、宇宙論的前提、境界論、仕事の定義、創作実践、位相非明示の提示方針、AI支援要請が、一つの流れの中に重ね書きされた自己記述文である。こうした文章では、何を言っているかだけでなく、どういう条件で読まれるべきかが重要になる。通常の要約は主題を抽出できても、提示プロトコルや位相差を落としやすい。そこで本試作では、文章を単に整理するのではなく、どの要素を保存対象と見なし、どの要素を変換可能とみなすかを分けて扱う必要があると考えた。

## 三モードの設計

`plain` は、基底理論を参照せず、入力文だけから骨格を整理するモードである。出力先は「一般に共有されている科学観・人間観・社会的常識に近い説明レジーム」に統一し、外部知識の注入を避ける。`basis_on` は、AMP / ITS / 履歴場通信論 / README などの理論ファイルを、出力先の世界観としてではなく、入力理解のための基底として参照するモードである。したがって、最終出力はなお常識的な説明レジームに置かれるが、入力文中の語彙や比喩は、基底語彙に沿って整列される。`bridge_on` は、そのうえでさらに、不変核を最大三つまで抽出し、何を守って運ぶかを明示した橋渡し変換を行うモードである。今回の比較では、この三モードの差を「出力先の思想差」ではなく、「どこまで理論を参照し、どこまで不変核を明示的に扱うか」の差として切り分けた。

## 結果と解釈

結果として、`plain` でもすでに核らしきものの抽出が起きていた。これは重要である。すなわち、通常の LLM も構造整理の過程で、ある種の bridge 的操作を暗黙に行っている可能性がある。もっとも、それはヒューリスティックな抽出であり、何を核とみなしたか、どこを保護すべきか、どこに誤読の危険があるかは十分に外在化されない。`basis_on` では、こうした暗黙操作が基底語彙によって安定化され、入力文の理解が深くなる。ただし、この段階でも保存対象と変換対象の境界はまだ曖昧である。これに対して `bridge_on` では、「越境可能な統合支援」「境界膜の定義・緩和・観測」「位相潜在化とAI整理」という三核が明示され、さらに保護境界・禁止読解・監査項目が付与された。したがって Bridge Lite の固有性は、橋渡し能力をゼロから発明することではなく、LLM に内在しうる暗黙の構造化傾向を、**再現可能で検査可能な手順へ押し出すこと**にあると考えられる。

## 限界

もっとも、本稿は完成理論の提示ではない。対象は単一ケースであり、しかも背後に比較的厚い基底理論群が存在していたため、そこで得られた三モード差をそのまま一般的な文章処理へ拡張することはできない。また、「目的・方法・提示プロトコル」という三核が普遍的な分解である保証もない。さらに、基底が強くなるほど `basis_on` と `bridge_on` の差は縮まりうる。今後は、基底を単に使う／使わないで切るのではなく、**どの程度の強さで効かせるか**、さらには **どの公理束を採るか** を可変にした比較が必要になる。公理選択憲章が示すように、思想は関心と捨象、価値とコスト、TPOとの関係で選ばれるため、基底は「唯一正しいOS」ではなく、選択可能な公理空間として扱う方が自然である。そうなると、文章転写の評価も「訳せたか否か」だけでは足りず、近接運用可否、公理衝突率、未翻訳残差、forced projection の有無、さらには何を見えやすくし何を見えにくくしたかというフィルター効果まで含めて考える必要がある。

## 今後の方向

今後の課題は三つある。第一に、複数タイプの文章を用いた比較である。高密度で意味が通った文だけでなく、擬似深遠文やジャーゴンサラダ、あるいは純粋な技術文書に対しても同様の比較を行い、核の捏造抑制や unknown の保持ができるかを確かめる必要がある。第二に、段落・文単位のタグ付けである。全文比較だけでなく、どの文がどの核を担っているかを局所的に追跡できれば、Bridge Lite は「うまくいった／いかなかった」の比較から一歩進める。第三に、オプションスイッチの設計である。基底強度、採用する公理束、未知の扱い、核優先度、フィルター効果の監査などを制御項目として導入することで、Bridge Lite は単なる文章変換器から、**基底選択的なブリッジ制御器**へ発展しうる。

## 結語

以上より、本稿の到達点は、問題を解き切ったことではなく、問題の形と最小構成を固定できたことにある。Bridge Lite はまだ最小構成にすぎないが、その最小構成が一つのケースに対して動作し、同時に次の拡張軸――基底強度、公理束選択、フィルター効果、近接運用可否――を明確に示したことは、小さくない成果である。本稿は、構造保存的文章転写と、その先の公理選択的ブリッジ設計に向かうためのアンカーとして位置づけられる。

**「同フォルダの JSON は raw log の未改変保存である」**
